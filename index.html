Contexte SACEM

La transformation digitale d'une société de gestion des droits d'auteur historique

La SACEM en chiffres

Fondée en 1851, la SACEM (Société des Auteurs, Compositeurs et Éditeurs de Musique) est une société de gestion collective des droits d'auteur qui représente plus de 200 000 membres (auteurs, compositeurs et éditeurs) et gère un répertoire de plusieurs millions d'œuvres.

D'histoire

Les défis technologiques

Face à la digitalisation accélérée du secteur musical et l'émergence de nouveaux usages, la SACEM a dû repenser son système d'information pour :

  • Moderniser l'infrastructure

    Passer d'une architecture monolithique à une approche microservices

  • Ouvrir l'écosystème

    Faciliter les échanges avec les partenaires et plateformes digitales

  • Accélérer l'innovation

    Réduire le time-to-market des nouvelles fonctionnalités

  • Gouverner les APIs

    Mettre en place une gouvernance robuste des échanges applicatifs

Contexte Initial

En 2018, la SACEM disposait d'une architecture on-premise traditionnelle avec des échanges applicatifs peu structurés et une gouvernance API limitée.

  • Architecture monolithique dominante
  • Peu d'APIs formalisées
  • Gouvernance ad-hoc

Objectifs

La DSI a défini une stratégie claire pour structurer et gouverner les échanges applicatifs avec WSO2 API Manager comme pierre angulaire.

  • Standardiser les échanges
  • Implémenter une gouvernance API
  • Préparer l'hybridation cloud

Approche

Une approche progressive a été adoptée, combinant transformation technique et organisationnelle pour assurer l'adoption.

  • Équipe transverse dédiée
  • Architecture évolutive
  • Intégration SAFe

Architecture & Innovations

Découvrez l'architecture technique et les innovations clés de la plateforme

Modèle Self-Service avec Gouvernance

La plateforme a évolué vers un modèle self-service permettant aux équipes métiers de publier leurs APIs tout en maintenant une gouvernance stricte.

Principes clés :
  • Autonomie contrôlée

    Les équipes peuvent publier des APIs via des workflows prédéfinis avec des étapes de validation automatiques ou manuelles

  • Portail intuitif

    Interface utilisateur simplifiée guidant les développeurs à travers le processus de publication

  • Templates standardisés

    Modèles pré-approuvés pour différents types d'APIs (REST, GraphQL, Événements)

Rôles et Permissions

Hiérarchie fine des droits d'accès avec 5 niveaux de permissions différents selon les profils utilisateurs.

  • API Publisher
  • API Creator
  • Admin
  • Ops
  • Consumer
Workflows

3 types de workflows configurés pour couvrir les différents scénarios de publication d'API.

  • Publication simple (automatique)
  • Publication avec validation technique
  • Publication avec validation métier
Sécurité

Mécanismes de sécurité intégrés pour garantir la conformité aux standards de la DSI.

  • Validation OWASP automatique
  • Scoring de qualité
  • Audit trail complet

Architecture Hybride On-Premise/Cloud

La plateforme combine intelligemment des composants on-premise et cloud pour optimiser performance, coût et conformité.

Composants clés :
  • On-Premise

    Composants critiques et données sensibles hébergés dans le datacenter de la SACEM

  • Cloud Public

    Composants scalables et interfaces externes hébergés sur AWS et Azure

  • Connectivité

    Réseau privé sécurisé avec connectivité directe aux fournisseurs cloud

Performances

Optimisation de la localisation des composants pour minimiser la latence :

  • Gateways déployées régionalement
  • Cache distribués
  • Load balancing intelligent
Latence réduite de 95%
Disponibilité 99.9%
Optimisation Coûts

Approche FinOps pour équilibrer performance et coût :

  • Auto-scaling des composants cloud
  • Réservation d'instances pour charges stables
  • Monitoring et alertes de coûts
Coûts réduits de 40%
Utilisation optimisée à 80%

Workflows Automatisés de Publication

Système de workflows configurables qui automatisent le processus de publication d'API tout en maintenant la gouvernance.

Étapes clés :
  1. 1
    Soumission

    Le développeur soumet l'API via le portail self-service avec les métadonnées requises

  2. 2
    Validation Automatique

    Tests de sécurité, qualité de code, conformité aux standards

  3. 3
    Approbation

    Validation technique et/ou métier selon le type d'API

  4. 4
    Publication

    Déploiement automatique sur les environnements cibles

Gains de Temps

Réduction drastique du temps de publication :

  • De 2 semaines à 2 heures pour les APIs simples
  • De 6 semaines à 1 semaine pour les APIs complexes
Configuration

Workflows entièrement configurables :

  • Conditions déclenchantes
  • Actions automatiques
  • Points d'approbation
  • Notifications

Intégration aux Pipelines CI/CD

Intégration profonde avec les outils CI/CD existants pour permettre une publication continue des APIs.

Flux d'intégration :
  • Jenkins

    Orchestration des pipelines de build et déploiement

  • Git

    Gestion du code source et déclenchement des pipelines

  • Artifactory

    Stockage des artefacts et gestion des dépendances

  • Kubernetes

    Orchestration des déploiements en production

Workflow Type

Exemple de workflow CI/CD pour une API :

  1. Commit dans Git
  2. Build et tests unitaires
  3. Analyse de sécurité
  4. Déploiement en dev
  5. Tests d'intégration
  6. Validation manuelle
  7. Déploiement en prod
Métriques

Impact de l'intégration CI/CD :

  • 80% des APIs avec déploiement continu
  • 50% de réduction des erreurs
  • 90% de tests automatisés
80% APIs en CI/CD
90% Tests Automatisés

Architecte système principal, SACEM
